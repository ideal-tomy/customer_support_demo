# Dual Lens Experience Plan

**Demo:** `customer-support`（`customer_support_demo`）  
**目的:** 実サービス相当の顧客体験と、自社導入を納得させるロジック解説を、同一デモで両立する。  
**前提:** Phase 0〜5（業種パック基盤）は完了済み。本 PLAN はその上に載せる次期大枠。

## 1. 目指す二つの感情

| レンズ | 体験者が思うこと | 主に触るもの |
|--------|------------------|--------------|
| **Experience（顧客層）** | 「ユーザーとして便利だ」 | チャット、ショートカット、回答カード、次アクション |
| **Adoption（導入層）** | 「自社・自店舗でも取り入れたい」 | デモ説明／仕組みカード、お試しナレッジ、Access Mode、3層の説明 |

同一人物が商談中に時間差で両方を味わう。**別アプリには分けない。**

---

## 2. 両立の原則

1. **デフォルトは Experience のみ**（Layer / fixture / LLM 等の技術語は出さない）
2. **Adoption 情報は明示トグルまたはシートでのみ増える**（既存デモ説明モードを拡張）
3. **同じ回答・同じナレッジ**を両層で見る（演出だけの別画面にしない）
4. **お試しナレッジは導入の本丸**として残し、導線を強化する
5. AI 接続は `@axeon/ai-demo-core` 経由（Provider 直叩き禁止、体験コードは Studio ポータルへ）

```text
┌─────────────────────────────────────┐
│ ヘッダー（店舗ブランド・AI対応中）     │  ← Experience
├─────────────────────────────────────┤
│ チャット（結論・CTA・常時ショートカット）│  ← Experience
│ 参照したご案内: …                      │  ← Experience（信頼）
│ ┄┄┄ Adoption ON のときだけ ┄┄┄        │
│ 仕組みメモ: 方式・根拠・次アクション型  │  ← Adoption
├─────────────────────────────────────┤
│ 入力欄 ＋ 常時ショートカット            │  ← Experience
├─────────────────────────────────────┤
│ ご案内 / 設定 / 有人                   │
│ お試しナレッジ・Access・デモ説明        │  ← Adoption 入口
└─────────────────────────────────────┘
```

## 3. 現状ギャップ（本 PLAN が埋めるもの）

| 領域 | 現状 | 目指す姿 |
|------|------|----------|
| 共通プロンプト | `common-prompt.md` がコード未連結 | 共通全文 ＋ スキーマ ＋ 業種 overlay |
| ショートカット | 初回・継続はあるが役割が曖昧／サンプル3往復上限 | 文脈CTA（A）と常時ショートカット（B）を分離し、自由入力後も B を維持 |
| 回答カード | 結論・箇条・根拠。条件／手順が未表示 | 条件・注意・主CTAまで顧客向けに表示 |
| 自由質問の次アクション | LLM 依存で不安定 | プロンプト契約 ＋ 業種デフォルトフォールバック |
| デモ説明 | 「回答方式」1行 | 仕組みカード（方式・根拠・次アクション型）＋ 3層パネル |
| 商談導線 | 口頭頼み | A→B→C（便利→仕組み→載せ替え）を UI で案内 |

## 4. Experience 層（顧客体験）

### 4.1 プロンプト3段

```text
[共通ベース] common-prompt.md 全文
  + [スキーマ契約] JSON / citations / escalate_human 等
  + [業種 overlay] 禁止事項・断定禁止・必須 followUps
```

業種で必ず残す差分:

| 業種 | 絶対ガード | 必須フォローアップ例 |
|------|------------|----------------------|
| 小売 | 在庫・着日断定禁止、返金確約禁止、配送現在地は番号 or 有人 | 申請方法 / 配送確認 / 担当者 |
| 飲食 | 空席断定禁止、アレルギー「完全安全」禁止 | 空席確認 / 予約 / 店舗相談 |
| エステ | 診断・効果断定禁止、痛み・赤みでメニュー提案禁止 | メニュー / 予約 / スタッフ（危険時は医療機関） |

自由質問向け共通契約:

1. 不足情報 → `missingInfoChoices` 優先  
2. Layer 3 → 確認操作 or 有人（断定しない）  
3. 解決時 → 業種主CTA ＋ 関連 0〜2 ＋ 有人を末尾候補  
4. 根拠なし → `not_found` ＋ 別用件 / 有人  

### 4.2 チップ二層

| 層 | 役割 | 配置 |
|----|------|------|
| **A. 文脈CTA** | この回答への一手 | 回答カード直下（FollowUp） |
| **B. 常時ショートカット** | 別用件の入口 | 入力欄上またはスレッド末尾（ルート固定、自由入力後も消さない） |

常時 B に置くのは入口として強いものに限定（業種別ルート）。ストーリー後半の質問は A（関連CTA）向き。

サンプルの往復上限（現状3）は、通常体験では緩和または撤廃。デモ説明用途に残す場合は Adoption / デモ説明 ON 時のみ制限する等を検討。

### 4.3 回答カード

顧客向け骨格:

1. 状態バッジ（顧客文言）  
2. 結論  
3. 要点（2〜4）  
4. 注意・条件（不利条件は省略しない）  
5. 次のアクション（主1＋副）  
6. 参考（折りたたみ、デフォルト閉じ）  

社内デモ由来の承認者・担当部署などは顧客UIに出さない。

### 4.4 CTA フォールバック

LLM が followUps 不足のとき、status / 業種デフォルトでチップを補完する（危険語時はエステでメニュー提案を出さない等）。

## 5. Adoption 層（導入ロジック）

### 5.1 既存資産（削除せず強化）

- デモ説明モード（`demo-explain-mode` / sessionStorage）
- 「参照したご案内」（顧客向け）と「回答方式」（技術向け）の分離
- お試しナレッジ（セッション内載せ替え）
- Access Mode（Sample / BYOK / Managed Trial）と Trial ポータル誘導
- 業種パック選択（差し替え証明）

### 5.2 仕組みカード（デモ説明 ON 時）

1回答につき最大3行:

| 行 | 伝えること |
|----|------------|
| どう答えたか | 固定 / AI＋ナレッジ / 業務連携（モック） |
| 何を根拠にしたか | 参照文書タイトル |
| 次に何をしたか | 提示したアクションの型（申請・確認・有人など） |

### 5.3 固定説明パネル

設定または Knowledge 付近に、短い常設説明:

- 回答3層（L1 / L2 / L3）の図または箇条
- 自社に入れる手順（サンプル → お試しナレッジ → Access Mode）
- Won't の明示（実注文・本番有人キューはデモ外）

### 5.4 商談フロー（プロダクトに埋め込む）

```text
Phase A  顧客になりきる（Adoption OFF）
  業種選択 → ショートカット／自由質問 → 結論と次アクション

Phase B  仕組みを見る（Adoption ON）
  同じ会話で仕組みカードを振り返る

Phase C  自社に載せる
  お試しナレッジ投入 → 同じ UI で質問
```

導線例:

- 1往復後、Adoption OFF なら「この案内の仕組みを見る」で ON
- Knowledge 先頭に「自社の案内で試す」

営業端末用に `?explain=1` 等で初回 Adoption ON も検討対象。

## 7. やらないこと（本 PLAN のスコープ外）

- 業種ごとの別リポジトリ・別アプリ化
- 本番在庫・予約・配送 API の本接続（モックのまま）
- 本番有人キュー / CRM 連携
- 顧客UIへの常時技術用語表示
- 導入専用の別 URL アプリ分割
- お試しナレッジ機能の削除・隠蔽
- デモ説明モードの削除

---

## 8. 成功条件（サマリ）

**Experience**

1. ショートカット／自由入力のどちらでも次の一手が分かる  
2. 不利条件が省略されず、必要なら有人へ自然に渡る  
3. 「また使いたいチャット」に見える  

**Adoption**

1. 同じ回答で固定 / 横断 AI / 業務モックの違いが言える  
2. 自社 FAQ 投入後、同じ UI で答えが変わる  
3. 業種パック差し替えと Core 接続が説明できる  

---

## 9. 推奨着手順（短い版）

1. **DL-0** 境界と受け入れを文書で固定する  
2. **DL-1** 顧客体験を先に上げる（プロンプト・ショートカット・回答カード）  
3. **DL-2** 既存デモ説明・お試しナレッジを仕組みカード／導線で強化する  
4. **DL-3** 営業台本と任意の URL 初期化  

Core 更新が必要な場合は `AI-Demo-Studio` でビルド後、本デモで `npm run vendor-core`。

## 10. 付録 — 常時ショートカット候補（B）

実装時の初期案。詳細は各業種 README と突合する。

| 業種 | 常時 B（例） |
|------|----------------|
| 小売 | 返品期限 / 配送状況 / 商品を選びたい / 返品・交換 |
| 飲食 | 営業時間 / コースを相談 / 空席・予約 / アレルギー |
| エステ | 初めてのメニュー / 予約・遅刻 / スタッフに相談 |

ストーリー用（返品×保証、今日19時4名、危険症状など）は **A（関連CTA）** 側に寄せる。
